Discover an unknown Bluetooth device

Web Bluetooth has no way to list multiple nearby devices at once the way a native scanner can -- requestDevice({acceptAllDevices: true}) is the standards-track, no-flag path: one real device per click, via the browser's own picker, not a live scanning list (the alternative, requestLEScan(), is a non-standard Chrome API gated behind an experimental flag). Click Scan, pick a device in the dialog that opens, and it's added to the list below. "Add to playlist" writes it into device-playlist.toml with no known service/characteristic yet -- add those by hand afterward if you want GATT reads, the same as any other Bluetooth playlist entry.

USB

Raw WebUSB -- no standardized interface to resolve a name against (unlike BLE GATT), so each entry names its own interfaceNumber and endpointNumber explicitly. One click performs one transferIn read; there's no generic push mechanism for raw USB, so re-click to read again.

HID (USB or Bluetooth)

WebHID reaches HID-class devices regardless of transport -- this is also the only way a browser can reach a Bluetooth HID device at all, since Web Bluetooth deliberately blocks the standard HID-over-GATT service. Live updates flow via the inputreport event, same one-click-then-live-updates shape as the BLE section.
